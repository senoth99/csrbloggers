# casher-bloggers-panel

Внутренняя CRM-панель для управления блогерскими интеграциями Casher Collection.
Сотрудники ведут контрагентов (блогеры), интеграции (выходы контента), доставки товаров, задачи, промокоды.

## Стек

- **Next.js 14** (App Router), **TypeScript**, **Tailwind CSS**
- **Prisma + SQLite** — хранилище пользователей и единый JSON-снапшот данных
- **Нет UI-библиотеки** — всё написано вручную на Tailwind
- **Нет тестов**

## Запуск

```bash
npm run dev  # predev сам создаст .env, пушнет БД, сделает seed
# Логин: senoth / admin (dev)
```

## Архитектура данных (ключевой момент)

Вся бизнес-логика — контрагенты, интеграции, доставки, сотрудники — живёт в **одной JSON-колонке** `PanelSnapshot.data` (SQLite).
Запись через `PUT /api/panel-data` с оптимистичной блокировкой по полю `revision` (заголовок `X-Panel-Base-Revision`).
На клиенте данные дополнительно кэшируются в `localStorage` под ключом `casher-panel-data-v1`.

Пользователи панели и сессии — отдельная таблица `User` в той же SQLite.

## Хеширование паролей

Новые пароли хешируются `scrypt(password, random_salt, 32, {N:16384,r:8,p:1})` и хранятся как `scrypt:<salt_hex>:<hash_hex>`.
Старые хеши (SHA-256, формат без префикса) верифицируются автоматически и **перехешируются при первом входе** — миграция прозрачная, без сброса паролей.

### 🔴 Баги (логика сломана)

**B1. Метрики дашборда считаются по `createdAt`, а не по `releaseDate`**
`integrationsPublishedInMonth` (`dashboard-metrics.ts`) фильтрует по `createdAt` — дате создания записи.
Если интеграция создана в апреле, а вышла в мае — в мае она не попадёт в KPI. Охваты, бюджет, CPM за месяц — всё считается неправильно.
**Фикс:** фильтровать `integrationsPublishedInMonth` по `releaseDate`, а не по `createdAt`.

**B2. Bar-chart охватов по дням тоже по `createdAt`**
`integrationReachByCalendarDayInMonth` (`dashboard-metrics.ts`) использует `createdAt` для определения дня.
Охваты отображаются в день добавления записи, а не в день выхода. При ретро-вводе данных — полная каша.
**Фикс:** использовать `releaseDate` вместо `createdAt`.

**B3. `isoInYearMonth` — UTC-баг**
`new Date(iso).getFullYear()` и `getMonth()` возвращают UTC-значения.
Пользователь в UTC+3 создаёт запись в 00:30 2 мая — UTC будет 1 мая, запись попадёт в апрельские метрики.
**Фикс:** использовать `getUTCFullYear`/`getUTCMonth` последовательно, или хранить даты как YYYY-MM-DD и разбирать локально.

**B4. `currentYearMonth()` замораживается при монтировании компонента**
В `DashboardScreen`: `const ym = useMemo(() => currentYearMonth(), [])`.
Если приложение открыто в конце месяца и пользователь не обновляет страницу — в новом месяце дашборд покажет старый месяц.
**Фикс:** либо `useState` с обновлением по таймеру, либо просто убрать `useMemo` (вычисление дешёвое).

**B5. Задача «убедиться в выходе интеграции» не создаётся без времени**
`buildOpenTasks` (`panel-tasks.ts`): `localReleaseDateTimeMs` возвращает `null` если нет `releaseTime`.
У интеграции может быть дата без времени — задача на проверку выхода не появится.
**Фикс:** если нет времени, брать начало дня как момент выхода.

**B6. `completedTaskKeys` в общем снапшоте — race condition между пользователями**
Когда двое отмечают разные задачи выполненными одновременно — один перезапишет изменения другого (409 конфликт ревизии). Второй попытки нет.
**Фикс:** хранить `completedTaskKeys` per-user (в таблице `User` или отдельной таблице).

**B7. Дельта промокодов = 0 в начале месяца**
`promoDeltaByCodeKey` в `DashboardScreen` считает `last - first` снапшот за текущий месяц.
В первый день месяца есть только один снапшот → `first === last` → дельта = 0.
Пользователи видят 0 активаций весь первый день.
**Фикс:** для первого снапшота месяца использовать 0 как базу (или последний снапшот прошлого месяца).

**B8. O(n²) при рендере списка контрагентов**
В `ContractorsScreen`: `rating10ByContractorId` внутри `useMemo` делает `integrations.filter(i => i.contractorId === c.id)` для каждого контрагента.
100 контрагентов × 1000 интеграций = 100 000 итераций при каждом рендере.
**Фикс:** предварительно группировать интеграции в `Map<contractorId, Integration[]>`.

### 🟠 UI/UX проблемы

**U1. Двойной `usePromocodes()` — двойной запрос к API**
`ContractorDetailScreen` и `DashboardScreen` оба вызывают `usePromocodes()` независимо.
При открытии карточки контрагента + дашборда — два параллельных запроса к `/api/promocodes`.
**Фикс:** вынести в `PanelDataContext` или отдельный `PromocodesContext`.

**U2. «Статкарта промокодов» и панель промокодов показывают разные данные**
Карточка KPI «Активаций промокодов» (`sumPromoActivations`) — сумма ручного поля `promoActivations` на интеграциях.
Панель промокодов ниже — данные из живого API Casher.
Пользователь видит два разных числа без объяснения что это разные источники.
**Фикс:** подписать источник данных, или убрать одно из двух.

**U3. Таблицы нескроллируемы на мобильном без подсказки**
Все таблицы имеют `min-w-[1100px]` и обёртку `overflow-x-auto`, но нет визуального индикатора горизонтального скролла.
На мобильном пользователь не понимает, что таблица продолжается вправо.
**Фикс:** добавить fade-right градиент или "→ прокрутить" при ширине < порога.

**U4. В карточке контрагента вкладки без счётчиков**
«Наличие», «Доставки», «Интеграции», «Ссылки» — не видно сколько элементов в каждой.
Пользователь не знает, в какую вкладку смотреть.
**Фикс:** добавить счётчик рядом с названием вкладки (как в Gmail).

**U5. Нет подтверждения при удалении**
Удаление контрагента, вещи, ссылки — без confirm-диалога. Жирная ошибка на один клик.
**Фикс:** добавить inline-подтверждение (двойной клик или mini-dialog).

**U6. Нет пустых состояний с подсказкой «что делать первым»**
Новому пользователю показывается пустой дашборд с нулями без объяснения.
**Фикс:** на пустом дашборде показывать onboarding-карточку со ссылками на «Создать контрагента» → «Создать интеграцию».

**U7. Путаница в именовании полей контрагента**
В форме создания: поле «ФИО» → сохраняется в `contactPerson`, поле «Никнейм» → в `name`.
В таблице колонка «ФИО» показывает `contactPerson || name`.
`name` — это технический ключ/никнейм, но везде отображается как основное имя. Непоследовательно.
**Фикс:** либо переименовать поле в интерфейсе, либо унифицировать отображение.

**U8. Маршрут `/panel/[integrationId]` — непонятное название**
«Панель» — это весь продукт. Но `/panel/[id]` — карточка конкретной интеграции.
Пользователи и разработчики путаются.
**Фикс:** переименовать маршрут в `/integrations/[id]`.

**U9. Нет focus-trap в модальных окнах**
При открытом модале пользователь может Tab-ом уйти «за» модал в фоновый контент.
Не соответствует WCAG 2.1 (criterion 2.4.3).
**Фикс:** добавить `focus-trap` при открытых модалах (пакет `focus-trap-react` или своя реализация).

**U10. Метки времени без таймзоны вводят в заблуждение**
`createdAt` хранится как `new Date().toISOString()` (UTC), но отображается через `formatRuDate` которая форматирует в локальной таймзоне браузера.
Для пользователей в разных таймзонах одна и та же запись показывает разные даты.
**Фикс:** использовать единую таймзону для хранения и отображения.

### 🟠 Архитектура

**A1. Весь домен в одном JSON-блобе**
`PanelSnapshot.data` — единый объект со всеми массивами.
Проблемы: нет нормальных запросов, нет индексов, конфликты при параллельных изменениях.
Решение: вынести каждый домен в отдельную Prisma-модель (`Contractor`, `Integration`, `Delivery`, …).

**A2. `PanelDataContext` — 1785 строк god-object**
Содержит: загрузку данных, синхронизацию с сервером, все CRUD-операции, бизнес-логику задач, промокодов и дашборда.
Нужно разбить на отдельные хуки/контексты: `useContractors`, `useIntegrations`, `useDeliveries`, `useTasks`.

**A3. Экраны смешивают логику и рендер**
`ContractorDetailScreen.tsx` — 1541 строк, `PanelScreen.tsx` — 1052, `DeliveriesScreen.tsx` — 907.
Нужно выносить бизнес-логику в хуки, компоненты дробить.
